# Cluster-Lab 部署阶段实战笔记

## 项目目标

在本地（macOS）用 Docker 搭建一个微型 HPC 集群，实现 Slurm 作业调度系统的完整部署与功能验证。

## 概述

本文档记录了部署过程中遇到的 11 个关键问题，涵盖 Docker 环境配置、依赖安装和 Slurm 服务调优。每个问题按现象、根因和修复方案三部分记录，供后续参考。

---

## 坑点记录

### 1. Docker Desktop 镜像源不可达

**现象：** `docker pull` 超时失败，无法拉取 Ubuntu 基础镜像。

**根因：** 宿主机 `~/.docker/daemon.json` 中配置了 USTC 镜像源（`docker.mirrors.ustc.edu.cn`），该镜像在丹麦网络环境下不可达。

**修复：** 编辑 `~/.docker/daemon.json`，移除 `registry-mirrors` 配置，使用 Docker 官方源。

```
// ~/.docker/daemon.json（移除前）
{
  "registry-mirrors": ["https://docker.mirrors.ustc.edu.cn"]
}

// 移除后
{}
```

**教训：** 镜像源配置因地理位置而异，macOS 上 Docker Desktop 的全局配置通过右上角 Docker 图标 → Settings → Docker Engine 修改，或直接编辑 `~/.docker/daemon.json`。

---

### 2. Docker credential store 缺失

**现象：** `docker pull` 报错 `docker-credential-desktop executable not found in $PATH`。

**根因：** macOS 上 Docker Desktop 使用 `credsStore: "desktop"` 管理凭据，但 `docker-credential-desktop` 二进制不在 shell 的 PATH 中。Docker 引擎启动时调用该凭据助手失败，导致拉取中断。

**修复步骤：**
1. 检查 `docker-credential-desktop` 实际位置（通常在 `/Applications/Docker.app/Contents/Resources/bin/docker-credential-desktop`）并创建符号链接到 `/usr/local/bin/`
2. 清空 `~/.docker/config.json` 中的 `credsStore` 字段，改用明文凭据存储

```
# 修复凭据助手路径
sudo ln -sf /Applications/Docker.app/Contents/Resources/bin/docker-credential-desktop /usr/local/bin/docker-credential-desktop

# 或临时清空 config.json
# ~/.docker/config.json
{}
```

**教训：** Docker Desktop 安装后可能需要手动修复 PATH。非交互式 shell（如脚本）中尤其容易出现此类问题。

---

### 3. 构建依赖缺失

**现象：** Docker 构建时 `apt install` 报错 `E: Package 'libhttp-parser-dev' has no installation candidate`。

**根因：** Ubuntu 24.04 的 apt 源中部分旧包名已变更，`libhttp-parser-dev` 和 `libjson-c-dev` 缺失。

**修复：** 修改 Dockerfile，替换为 Ubuntu 24.04 对应的包名，并添加 `apt update` 确保源列表最新。

```dockerfile
# 修复前（老旧包名）
RUN apt install -y libhttp-parser-dev libjson-c-dev

# 修复后
RUN apt-get update && apt-get install -y \
    libhttp-parser-dev \
    libjson-c-dev
```

**教训：** 依赖 lib 的包名在不同 Ubuntu 版本间可能变化，构建前应先验证 apt 源中是否包含所需包。

---

### 4. Munge 启动方式变更

**现象：** `service munge start` 报错 `munge: unrecognized service`。

**根因：** Ubuntu 24.04 使用 systemd 管理服务，不再提供 `/etc/init.d/munge` 脚本。Munge 需要手动创建密钥并以后台进程方式启动。

**修复：** 修改 `entrypoint.sh`，将 Munge 启动改为两步：

```bash
# 修复前（Ubuntu 22.04 方式）
service munge start

# 修复后（Ubuntu 24.04 方式）
# 1. 创建 Munge 密钥（仅首次需要）
create-munge-key -f /etc/munge/munge.key

# 2. 修复权限
chown -R munge:munge /etc/munge /var/log/munge /var/run/munged
chmod 400 /etc/munge/munge.key
chmod 700 /var/log/munge

# 3. 后台启动 munged
munged --foreground &
sleep 1
```

**教训：** Ubuntu 24.04 LTS 移除了 sysvinit 脚本，所有服务的启动方式需要适配。Slurm 文档中的示例代码通常基于 CentOS/RHEL 或旧版 Ubuntu，不能直接照搬。

---

### 5. Slurm 配置兼容性问题

**现象：** `slurmctld` 启动后退出或报 `fatal: ... unknown configuration parameter`。

**根因：** Ubuntu 24.04 自带 Slurm 24.11，某些配置项在较新版本中已被废弃或改名。

**修复：** 逐一排查 `slurm.conf` 中的配置项：

| 问题配置 | 原因 | 修复 |
|---------|------|------|
| `AuthType=auth/munge` | Munge 在容器间认证不稳定 | 保留但配合 `AccountingStorageType=none` 绕过 dbd 认证 |
| `AccountingStorageType=accounting_storage/slurmdbd` | 需要完整 Munge 认证链 | 改为 `accounting_storage/none` 简化 |
| `ProctrackType=proctrack/cgroup` | Docker 中 cgroup v2 不可用 | 改为 `proctrack/linuxproc` |
| `CgroupPlugin` 相关 | cgroup v2 路径差异 | 显式指定 `CgroupPlugin=cgroup/v1` |

**教训：** Slurm 配置高度依赖版本和环境，官方文档示例多基于 CentOS 7/RHEL，在 Ubuntu + Docker 环境中需要大量调整。建议先禁用所有高级特性，验证基础调度可用后再逐步开启。

---

### 6. slurmdbd.conf 权限问题

**现象：** `slurmdbd` 启动时报 `Permission denied` 无法读取配置文件。

**根因：** `slurmdbd.conf` 必须由 slurm 用户所有，且权限必须为 600（不可被其他用户读取）。

**修复：**

```bash
chown slurm:slurm /etc/slurm/slurmdbd.conf
chmod 600 /etc/slurm/slurmdbd.conf
```

**教训：** Slurm 安全机制严格限制 `slurmdbd.conf` 的权限（含数据库密码），权限不符合要求直接拒绝启动。`slurm.conf` 则没有这么严格的要求。

---

### 7. Docker 内 Munge 认证断裂

**现象：** `sinfo` 报 `slurmctld: error: Munge authentication failed`，集群节点间无法互信。

**根因：** 容器环境下，Munge 认证依赖容器间共享的 `munge.key` 文件和时间同步。但 Docker 容器时钟可能存在微小偏差，且多个进程同时访问同一密钥文件可能产生竞态条件。

**修复方案（二选一）：**

**方案 A（推荐）：绕过 Munge 认证**
```conf
# slurm.conf
AccountingStorageType=accounting_storage/none
# 关闭数据库记录，slurmctld 不再需要与 slurmdbd 进行 Munge 认证
```

**方案 B（完整认证）：** 保留 Munge，但需确保：
- 所有容器共享同一 `munge.key`（通过 Docker volume）
- 容器间时间同步（NTP 或 Docker 主机时间挂载）
- munged 在所有节点上先启动，slurmctld/slurmd 后启动

**最终选择：** 开发/测试环境下使用方案 A 简化认证。生产环境应使用方案 B。

---

### 8. cgroup 兼容性（排查耗时最长的问题）

**现象：** slurmd 启动后立即退出，日志显示 `cgroup namespace` 相关错误。

**根因：** Docker 容器的默认 cgroup 模式是 v2，但容器的 cgroup namespace 隔离导致 Slurm 无法正确访问宿主机 cgroup 层级。slurmd 试图操作 cgroup 时因权限不足或路径不存在而崩溃。

**修复（三管齐下）：**

1. **slurm.conf 中切换到 linuxproc 跟踪**
```conf
ProctrackType=proctrack/linuxproc
# 不使用 cgroup 跟踪进程，改用 Linux 原生的 /proc
```

2. **禁用 cgroup 插件**
```conf
# 不设置 CgroupPlugin，或显式指定 v1
CgroupPlugin=cgroup/v1
```

3. **清空 cgroup.conf**
```ini
# cgroup.conf — 清空全部内容（Docker 环境不需要）
```

**教训：** 这是部署过程中需要最多调试步骤才能定位的问题。slurmd 无声退出，没有明显错误日志。最终通过加 `-vvvvv` 调试级别启动 slurmd 才看到 cgroup namespace 错误。Docker 容器中运行 Slurm 的最佳实践是全部禁用 cgroup 相关配置。

---

### 9. Spool 目录缺失

**现象：** slurmd 启动失败，日志显示 `fatal: can't create spool directory /var/spool/slurmd`。

**根因：** Docker 镜像中 `/var/spool/` 下没有 slurmd 需要的子目录。

**修复：** 在 `entrypoint.sh` 中启动服务前创建所需目录：

```bash
mkdir -p /var/spool/slurmd /var/log/slurm
chown -R slurm:slurm /var/spool/slurmd /var/log/slurm
```

同时确保这些目录持久化在 Docker volume 上（`docker-compose.yml` 中已配置）。

**教训：** Slurm 服务对目录结构的依赖很强，缺少任何一个都会导致启动失败。建议在 entrypoint 中统一创建并修复所有需要目录的权限。

---

### 10. NFS 挂载锁问题

**现象：** `mount -t nfs` 卡住超时，node01/node02 无法挂载 slurmctl 导出的 `/home`。

**根因：** Docker 容器默认不运行 `rpc.statd`（NFS 锁管理守护进程），NFS v3 的 `lockd` 机制在容器中无法正常工作，导致 `mount` 命令在等待锁服务时永久阻塞。

**修复：** 在 NFS 挂载命令中添加 `-o nolock` 选项，禁用文件锁：

```bash
# 修复前
mount slurmctl:/home /home

# 修复后
mount -t nfs -o nolock slurmctl:/home /home
```

**教训：** Docker + NFS 是一个常见但不完美的组合。NFS v3 的锁机制依赖内核模块 `lockd` 和用户空间服务 `rpc.statd`，这两者在容器中默认不可用。在测试环境中使用 `nolock` 是最实用的解决方案。

---

### 11. 服务启动顺序问题

**现象：** slurmctld 启动时找不到 slurmdbd，或 slurmd 启动时找不到 slurmctld。

**根因：** `entrypoint.sh` 中所有服务同时并行启动（`&`），没有等待依赖服务就绪。

**修复：** 在 `entrypoint.sh` 中实现依赖等待逻辑：

```bash
# 启动顺序（每个步骤等待前一步确认就绪）
# Step 1: MariaDB → slurmdbd → slurmctld（控制节点）
# Step 2: slurmd（计算节点，等待控制节点就绪）

# 控制节点启动序列
mysqld_safe &
# sleep 等待 MariaDB 就绪
sleep 5

slurmdbd -D -vv &
# sleep 等待 slurmdbd 就绪
sleep 2

slurmctld -D -vv &
# sleep 等待 slurmctld 就绪
sleep 2

# 计算节点启动序列
slurmd -D -vv &
```

**教训：** 多服务容器的启动顺序非常重要。使用固定的 `sleep` 间隔虽然不够精确但有效。生产环境应考虑使用更优雅的健康检查（如 `slurmctld -V` 或端口探测）。

## 经验总结

### 十大坑点分布

| 类别 | 坑点数 | 涉及 |
|------|--------|------|
| Docker 环境 | 2 | 镜像源、credential store |
| 操作系统差异 | 2 | Munge 启动方式、Ubuntu 24.04 包变更 |
| 配置兼容性 | 3 | Slurm 配置项、cgroup、slurmdbd 权限 |
| 容器特性 | 3 | Munge 认证断裂、NFS 锁、Spool 目录 |
| 编排问题 | 1 | 服务启动顺序 |

### 核心原则

1. **最小可行优先**：先用最简配置跑通，再逐步开启高级功能。不要一开始就配置完整的 Munge + cgroup + accounting。
2. **日志是王道**：Slurm 服务的调试日志非常详细，使用 `-vvvvv` 级别可以看到几乎所有问题。容器用 `docker logs` 查看。
3. **容器思维**：Docker 容器不是完整虚拟机，很多系统服务（systemd、NFS lockd、cgroup 控制器）不可用或受限。需要针对容器环境调整服务配置。
4. **环境隔离**：宿主机配置（Docker 镜像源、PATH、凭据助手）可能影响容器构建和运行。在文档中记录这些环境依赖。

### 最终架构

经过全部修复后，三层服务正常运行：

```
┌──────────────────────────────────────────────────────────────┐
│                    macOS 宿主机                              │
│  Docker Desktop (通过桥接网络 cluster-net)                    │
│                                                              │
│  ┌──────────┐      ┌──────────┐  ┌──────────┐              │
│  │ slurmctl  │      │  node01  │  │  node02  │              │
│  │ (管理节点) │      │ (计算节点)│  │ (计算节点)│              │
│  ├──────────┤      ├──────────┤  ├──────────┤              │
│  │ slurmctld│      │  slurmd  │  │  slurmd  │              │
│  │ slurmdbd │      └──────────┘  └──────────┘              │
│  │ MariaDB  │           ↑              ↑                    │
│  │ NFS 服务  │──────────┴──────────────┘                    │
│  └──────────┘     NFS 挂载 /home (nolock)                   │
└──────────────────────────────────────────────────────────────┘
```
