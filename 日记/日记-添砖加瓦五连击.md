## 五项功能全面实现

### 1. sbatch 作业模板

Slurm 的标准提交方式是 `sbatch`，而不是交互式的 `srun`。为了让集群具备完整的作业提交流程，创建了三个文件：

**hello_mpi.sbatch** — 跨节点 MPI 批处理：
```bash
#!/bin/bash
#SBATCH --nodes=2
#SBATCH --ntasks=2
#SBATCH --ntasks-per-node=1
#SBATCH --mpi=pmix
#SBATCH --time=00:05:00

module load gcc/13.3.0 openmpi/4.1.6
srun /home/hello_mpi
```

**hello_openmp.sbatch** — OpenMP 多线程批处理：
```bash
#SBATCH --nodes=1
#SBATCH --ntasks=1
#SBATCH --cpus-per-task=4
export OMP_NUM_THREADS=${SLURM_CPUS_PER_TASK:-4}
```

**scripts/jobs/README.md** — 使用说明。

使用方法：
```bash
docker exec slurmctl sbatch /home/scripts/jobs/hello_mpi.sbatch
docker exec slurmctl squeue      # 查看队列
```

### 2. test-cluster.sh 全面验证

这是一个 11 项自动验证脚本，一条命令跑完所有集群检查：

| 检查项 | 测试内容 |
|--------|---------|
| 1 | sinfo 节点注册状态 |
| 2 | srun hostname 单节点 |
| 3 | srun -N 2 跨节点可达性 |
| 4 | NFS 跨节点读写 |
| 5 | mpicc 编译 MPI 程序 |
| 6 | srun --mpi=pmix 跨节点 MPI |
| 7 | gcc -fopenmp 编译 OpenMP 程序 |
| 8 | OpenMP 多线程执行 |
| 9 | Lmod module load |
| 10 | srun 下 module 传递 |
| 11 | sbatch 作业提交 |

每条测试标记 PASS / FAIL，最后汇总。失败时输出具体哪一项有问题，方便定位。

```bash
docker exec slurmctl bash /home/scripts/test-cluster.sh
```

### 3. 多用户环境

创建了三个用户，对应三种不同的使用场景：

| 用户 | UID | 默认 QoS | 角色 |
|------|-----|---------|------|
| alice | 2001 | high | 特权用户，跑高优先级测试 |
| bob | 2002 | normal | 标准用户 |
| charlie | 2003 | low | 低优先级用户，适合批处理 |

**setup_users.sh** 实现了：
1. useradd 创建系统用户
2. sacctmgr 添加 Slurm 账户和用户关联
3. QoS 按用户分配

使用方法：
```bash
docker exec slurmctl bash /home/scripts/setup_users.sh
docker exec -u alice slurmctl srun hostname
```

注意：由于当前集群 `AccountingStorageType=none`，sacctmgr 的部分功能受限（如作业计费）。但账户结构和 QoS 分配是完整的，为后续启用计费做好了准备。

### 4. QoS 优先级配置

三种服务质量级别：

| QoS | 优先级 | 最大运行时间 | 最大作业数 |
|-----|-------|------------|-----------|
| high | 1000 | 1 天 | 10 |
| normal | 500 | 2 天 | 20 |
| low | 100 | 7 天 | 50 |

**setup_qos.sh** 使用 sacctmgr 创建 QoS 定义并分配给用户：

```bash
sacctmgr -i add qos high priority=1000 maxwall=1-00:00:00 maxjobs=10
sacctmgr -i modify user alice set qos=high
```

使用方法：
```bash
docker exec slurmctl bash /home/scripts/setup_qos.sh
docker exec slurmctl srun --qos=high -N 2 hostname
```

### 5. GRES GPU 模拟

GRES（Generic Resource Scheduling）是 Slurm 管理非通用资源（如 GPU）的机制。在没有物理 GPU 的 Mac 上模拟了 GPU 设备：

**gres.conf：**
```
NodeName=node01 Name=gpu File=/dev/fake_gpu[0-3] Count=4
NodeName=node02 Name=gpu File=/dev/fake_gpu[0-3] Count=4
```

**check_gpu.c** — 验证 GRES 分配的测试程序：
```c
// 检查 SLURM_JOB_GRES 环境变量
// 扫描 /dev/fake_gpu* 设备
// 报告 GPU 分配结果
```

**entrypoint.sh** 更新：容器启动时自动创建 4 个 fake GPU 设备：
```bash
mknod /dev/fake_gpu0 c 10 20
chmod 666 /dev/fake_gpu0
```

使用方法：
```bash
srun --gres=gpu:1 /home/check_gpu
# 或者用 ls 验证
srun --gres=gpu:1 ls /dev/fake_gpu*
```

## 附带修复

借着这次大改的机会，发现了几个历史遗留问题一并修复：

### IP 地址修正

最初的 slurm.conf 和 entrypoint.sh 中写死了旧的 controller IP（172.20.0.2），实际部署使用 172.20.0.10。三个地方全部修正：
- `SlurmctldHost=slurmctl(172.20.0.2)` → `172.20.0.10`
- NFS mount `172.20.0.2:/home` → `172.20.0.10:/home`
- NFS mount `172.20.0.2:/scratch` → `172.20.0.10:/scratch`

### Lmod 自动加载

entrypoint 中加入：
```bash
if [ -f /usr/share/lmod/lmod/init/bash ]; then
  source /usr/share/lmod/lmod/init/bash
fi
```
这样容器启动后无需手动 `source`，`module` 命令立即可用。
