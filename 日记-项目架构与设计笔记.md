# Cluster-Lab 项目架构与设计笔记

## 项目目标

在本地（macOS）用 Docker 搭建一个微型 HPC 集群，学习 Slurm 作业调度系统的架构、配置与日常管理。

## 架构说明

### 节点角色

| 容器名 | 角色 | 功能 |
|--------|------|------|
| slurmctl | 管理节点 | slurmctld（主调度器）+ slurmdbd（数据库）+ NFS server |
| node01 | 计算节点 | slurmd（执行守护进程） |
| node02 | 计算节点 | slurmd（执行守护进程） |

所有节点通过 Docker 自定义网络 `cluster-net` 互通，使用 Ubuntu 24.04 基础镜像。

### NFS 共享

- slurmctl 运行 NFS server，导出 `/home` 目录
- node01/node02 通过 NFS v3/v4 挂载 `/home`
- 实现用户主目录跨节点统一

## 设计决策与原因

### 1. 为什么用 Docker 而不是虚拟机？

- **资源消耗低**：宿主机共享内核，三节点约 1.5 GB 内存，比 VM 方案轻量得多
- **启动快**：秒级启动，便于反复测试、重建
- **OCI 标准**：镜像可打包分发，任何 Docker 环境都能复现环境

### 2. 为什么三个节点用同一份 Dockerfile？

- 基础环境高度一致：所有节点都需要 `munge + Slurm + NFS 客户端`
- 只需要控制容器进程差异化：`entrypoint.sh` 根据环境变量 `NODE_ROLE` 决定启动 slurmctld 还是 slurmd
- 减少镜像数，避免维护多份 Dockerfile 的重复劳动

### 3. 为什么不用 systemd 管理服务？

- Docker 容器内不建议运行 systemd（需要特权模式且违反单一进程理念）
- 替代方案：`entrypoint.sh` 用后台进程方式逐个启动（用户 `uucp` 启动 `slurmd`，用户 `slurm` 启动 `slurmctld`）
- 简化调试：用 `docker logs` 就能看到服务启动日志

### 4. 为什么用 init.d 脚本风格的 entrypoint？

- Docker 原生不支持容器内的守护进程管理
- 每个进程用 `&` 后台运行 + `wait` 等待所有子进程
- 实现"一个容器跑多个进程"的效果，同时保持容器的正常生命周期

### 5. 为什么 Slurm 配置用 ubuntu 系列节点名？

- Docker 容器的 hostname 默认派生自容器名，保持一致即可
- 配置中不依赖 IP 地址（IP 可能随重启变化），只使用 hostname

### 6. 为什么用 NFS 而不是其他共享方案？

- Linux 原生支持，配置成熟稳定
- 不需要额外软件栈（如 GlusterFS/Lustre 等分布式文件系统）
- 单管理节点场景下，NFS 足够简单高效

### 7. 为什么 Ubuntu 24.04？

- LTS 版本，支持周期长（2029 年）
- 官方仓库自带 Slurm 24.11 包，无需手动编译
- APT 包管理成熟，依赖自动解决

## Slurm 关键配置说明

`slurm.conf` 中最核心的部分：

- **`SlurmctldHost=slurmctl`**: 指定调度器所在节点，必须与 hostname 一致
- **`NodeName=node[01-02] CPUs=2 RealMemory=2000`**: 定义计算节点资源（2 核 CPU，2 GB 内存）
- **`PartitionName=debug Nodes=node[01-02] Default=YES`**: 单一分区 `debug`，方便测试作业
- **`SelectType=select/cons_tres`**: 基于 TRES（trackable resources）的资源选择，支持精细分配
- **`ProctrackType=proctrack/linuxproc`**: 轻量级进程跟踪，适合 Docker 环境（不需要 cgroup v2 依赖）
